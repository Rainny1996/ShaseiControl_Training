## 二、缺陷清单

### 🔴 致命（阻塞编译/运行时崩溃）

#### BUG-CT-01：`configureProtection()` 在 `onAppear` 中调用导致双重注册

| 文件 | ControlTrainingApp.swift:14 | ContentView.swift |
|------|---------------------------|------|
| 等级 | 致命 | 严重 |

```swift
// ControlTrainingApp.swift:14
.onAppear {
    SecurityService.shared.configureProtection()  // ← 每次 ContentView 出现都注册
    initializeDataIfNeeded()
}
```

**问题**：`configureProtection()` 在 `onAppear` 中调用，意味着每次 `ContentView` 重新出现（如从后台返回）都会重复调用 `NotificationCenter.addObserver`。同时 `AppDelegate.applicationDidEnterBackground` 已经调用了 `lockApp()`（L19），导致后台锁定被**双重触发**。

**定位**：
- `ControlTrainingApp.swift:14-16` — 应在 `init()` 中调用一次
- `AppDelegate.swift:17-19` — `didEnterBackground` → `lockApp()` ✅
- `SecurityService.swift:235-249` — `configureProtection()` 内部再次注册 `didEnterBackgroundNotification` → 与 AppDelegate 重复

**修复方向**：删除 `SecurityService.configureProtection()` 中的 `didEnterBackgroundNotification` 观察者（L235-241），或在 `ControlTrainingApp` 的 `init()` 中调用一次（配合 `@UIApplicationDelegateAdaptor` 的生命周期）。

### 🟠 严重（功能异常或数据风险）

#### BUG-CT-02：`deleteAllUserData()` 批量删除后 viewContext 缓存未刷新

| 文件 | CoreDataStack.swift:71-86 |
|------|--------------------------|

```swift
func deleteAllUserData() {
    let entities = ["TrainingRecord", "CheckInRecord", ...]
    performBackgroundTask { context in
        for entityName in entities {
            let request = NSFetchRequest<NSFetchRequestResult>(entityName: entityName)
            let deleteRequest = NSBatchDeleteRequest(fetchRequest: request)
            try? context.execute(deleteRequest)   // ← 批量删除绕过 viewContext
        }
    }
    // ❌ 未调用 container.viewContext.refreshAllObjects()
    // ❌ 未调用 container.viewContext.reset()
}
```

**问题**：`NSBatchDeleteRequest` 直接操作 SQLite，不通过 `viewContext`。删除后 `viewContext` 中仍持有已删除对象的缓存引用，访问它们会导致 `CoreData` 异常。必须调用 `container.viewContext.reset()` 或监听 `NSManagedObjectContextDidSave` 通知。

**修复**：在 `performBackgroundTask` 保存后，切回主线程调用 `container.viewContext.reset()`。

#### BUG-CT-03：`SecurityService` 与 `CryptoService` 使用独立的加密密钥

| 文件 | SecurityService.swift:334 | CryptoService.swift:17 |
|------|--------------------------|----------------------|

```swift
// SecurityService.swift:334 — 密钥名: "encryptionKey"
let keyKey = "encryptionKey"
let key = SymmetricKey(size: .bits256)

// CryptoService.swift:18 — 密钥名: KeychainKey.encryptionKey.rawValue
let keyKey = KeychainService.KeychainKey.encryptionKey.rawValue
let newKey = SymmetricKey(size: .bits256)
```

**问题**：两个服务各自生成独立的 AES-256 密钥存入 Keychain，且 key 名不同。`SecurityService.encryptData()` 加密的数据无法被 `CryptoService.decrypt()` 解密，反之亦然。如果 Repository 层使用了其中一个加密，而 ViewModel 层使用了另一个解密，将产生静默的解密失败（返回 `nil`）。

**当前影响**：检查代码后确认 Repository 层均使用 `CryptoService`，ViewModel 层也引用 `CryptoService`，`SecurityService` 的加解密方法未被外部调用——目前未触发但存在后继误用风险。建议统一为一个加密入口。

### 🟡 中等（代码质量问题）

#### BUG-CT-04：`DataController` 文件保护重复设置

| 文件 | CoreDataStack.swift:19-22 |
|------|--------------------------|

```swift
description.setOption(true as NSNumber,                          // ← 覆盖
                     forKey: NSPersistentStoreFileProtectionKey)
description.setOption(FileProtectionType.complete as NSNumber,   // ← 最终生效
                     forKey: NSPersistentStoreFileProtectionKey)
```

第二次 `setOption` 覆盖了第一次，最终文件保护级别为 `complete`。但第一行是多余的，且 `true as NSNumber` 不是有效的文件保护值。

---

#### BUG-CT-05：`TrainingRecord.selfRating` 无边界校验

| 文件 | TrainingModels.swift:148,157 |
|------|---------------------------|

```swift
let selfRating: Int
init(... selfRating: Int = 3, ...)
```

自评分数没有 1-5 的范围校验。可以传入任意 `Int` 值，导致分析服务中的评分计算失真。

**修复**：在 init 中添加 `precondition(selfRating >= 1 && selfRating <= 5)` 或使用 `clamped` 逻辑。

---

#### BUG-CT-06：AppDelegate 音频会话配置缺少 `allowBluetooth` 的 `mixWithOthers`

| 文件 | AppDelegate.swift:37-39 |
|------|------------------------|

```swift
try AVAudioSession.sharedInstance().setCategory(.playback,
    mode: .spokenAudio,
    options: [.duckOthers, .allowBluetooth])
```

与 `AudioService.swift:123-125` 重复配置音频会话，且 `AudioService.configureBackgroundPlayback()` 缺少 `.allowBluetooth`。两个入口设置不同的 options，后调用者覆盖先调用者。
